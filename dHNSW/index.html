dHNSW, Efficient Vector Search on Disaggregated Memory

Overview

  Disaggregating is gaining attention in cloud computing by separating storage and compute hardware resources, allowing them to scale independently for better flexibility and efficiency. Recent industry efforts, such as DeepSeek’s 3FS, have demonstrated the benefits of leveraging RDMA(Remote Direct Memory Access) for AI training and inference, enabling high-spped remote memory access with low latency.
  We introduce d-HNSW, the first vector search engine for RDMA-based disaggregated memory systems, which incorporates three key techniques:

  1. Representative index caching: construct a lightweight index from a sampled subset of the data and caches it in the compute pool to minimize frequent access to critical components of the hierarchical graph index
  2. RDMA-friendly data layout: optimze data placement to reduce networking round trips for both vector queries and insertions
  3. Batched query-awared data loading: mitigates bandwidth usage between memory and compute nodes

Details

Architecture

  d-HNSW exploites the characteristices of EDMA-based memory data accessing and graph-based index HNSW to realize fast and bandwidth-efficient vector query processing. d-HNSW achieves so representative index caching, RDMA-friendly graph index storage in remote memory, and query-aware batched data loading. Here, we provide a brief overview of d-HNSW:

Representative index caching

  Graph-based vector search schemes rely on greedy routing to iteratively toward the queried vector. However, the search path can span the entire graph, potentianlly covering distanct vectors and loading the entire graph index from memory pool to the computer pool for each query is impratical, because the compute pool has limited storage resources in a disggregated system. This approach would not only consume excessive bandwidth by transferring a significant portion of untraversed vectors but also introduce additional latency, thereby degrading the overall search efficiency.
  We propose partitioning the vector database into multiple subsets. Inspired by Pyramid, we construct a three-layer representative HNSW, referred to as meta-HNSW, by uniformly selecting 500 vectors. This meta-HNSW serves as lightweight index and a cluster classifier for the entire dataset.

  The search process starts from a fixed entry point in the top layer L2L_{2}L2​ of meta-HNSW and applies greedy routing at each layer, traversing downward until reaching a vector in its bottom layer L0L_{0}L0​. Each vector in L0L_{0}L0​ defines a partition and serves as an entry point to a corresponding sub-HNSW.
  The overall graph index consists of two components: meta-HNSW, which provides coarsed-grained classfication, and sub-HNSWs, which enable fine-grained search within partitions. To improve search efficiency in disaggregation, we cache the lightweight meta-HNSW in the compute pool, allowing it to identify the most relevant sub-HNSW clusters for a given query.

RDMA-friendly graph index storage layout in remote memory

  Given that the top-m closest sub-HNSW clusters for a queried vector qqq are S0,...,Sm−1{S_{0},...,S_{m-1}}S0​,...,Sm−1​, the compute instance can issue RDMA_READ commands to access these serialized clusters and then deserialize them. However, two challenges arises:

  1. If the required clusters S0,...,Sm−1{S_{0},...,S_{m-1}}S0​,...,Sm−1​ are not stored contiguously in memory, multiple RDMA round trips are required, increasing latency
  2. When new vectors are inserted, the size of each sub-HNSW cluster may exceed the allocated space. Since shifting all stacked sub-HNSW clusteres is impratical, newly inserted vectors and their matadata may be placed in non-contiguous memory regions if they are simply appended at the tail of the available area. This gragmentation increases access latancy and reduces query throughput due to the higher cost of scattered index access.
      As shown in Fig.4, we allocated and register a continuous memory space in memory instance to store both the serialized HNSW index and floating-point vectors. At the beginning of the meory space, a global metadata block records of the offsets of each sub-HNSW cluster, as their sizes vary. The remaining memory space is divided into groups, each of which is capable of holding two sub-HNSW clusters. Within each group, the first section stores the first serialized sub-HNSW cluster, which includes its metadata, neighbor array for HNSW, and the associated floating-point vectors. The second sub-HNSW is placed at the end of the group. Between these two clusters, we allocate a shared overflow memory space to accomdate newly inserted vectors dor both sub-HNSW clusters.

  When a vector query requires loading a sub-HNSW cluster, the compute instance issues an RDMA_READ command to retrieve the cluster along with its corresponding shared overflow memory space. This layout ensures that newly inserted vectors are stored continously with the original sub-HNSW data, enabling them to be read back with a one-time RDMA_READ command. To optimize memory usuage, each pair of adjacent sub-HNSW clusters shares a single overflow mmeory space for accomodating newly inserted vectors rather than allocating a separated one for each clsuter.
  If multiple sub-HNSW clusters nned to loaded into the compute pool for batched query processing, and they are not stored continuously in memory, we leverage doorbell batching to read them in a single network round-triip with RDMA NIC issuing multiple PCIe transactions.

Qeury-aware bacthed data loading

  To reduce bandwidth usuage for transferring graph index and improve query efficiency, we propose merging sub-HNSW index loading for queried vectors in the same batch. Given a batch of queried vectors q1,q2,...,qs{q_{1},q_{2},...,q_{s}}q1​,q2​,...,qs​ and a total of mmm sub-HNSW clusters, each queried vector requires seaching the top-k closest vectors from the bbb closest sub-HNSWs. However, the DRAMS resources in the compute instance can only accomdate and cache ccc sub-HNSWs. To optimize loading, we analyze the required b×sb \times sb×s sub-HNSWs online and ensure that each sub-HNSW is loaded from the memory pool only once.

  Once all required sub-HNSW clusters for the batched queried vectors have been loaded and traversed, the query results will be returned. Additionally, we retain the most recently loaded ccc sub-HNSWs for the next batch. If the required sub-HNSWs are already in the compute instance, they do not need to be loaded again, further reducing data transfer overhead.
