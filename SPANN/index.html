SPANN, Highly Efficient Billion Scale ANNS

Overview

  In this paper, we present a simple but efficient memory-disk hybrid indexing and search system, named SPANN, thta follows the inverted index methodology. It stores the centroid points of the posting lists in the memory and the large posting lists in the disk.

Details

  Index structure: the data vectors XXX are divided into NNN posting lists X1,X2,...,XN{X_{1}, X_{2},..., X_{N}}X1​,X2​,...,XN​, X1∪X2∪...∪XN=XX_{1} \cup X_{2} \cup ... \cup X_{N} = XX1​∪X2​∪...∪XN​=X. The centroids of these posting lists, c1,c2,...,cNc_{1},c_{2},...,c_{N}c1​,c2​,...,cN​ are stored in the memory as fast coarse-grained index that point to the location of the corresponding posting lists in the disk.
  Partial search: when a query qqq comes, we find the KKK closest centroids, ci1,ci2,...,cik,k<<N{c_{i1},c_{i2},...,c_{ik}}, k << Nci1​,ci2​,...,cik​,k<<N, and load the vectors in the posting lists Xi1,Xi2,...,XikX_{i1},X_{i2},...,X_{ik}Xi1​,Xi2​,...,Xik​ that correspond to the closest KKK centroids into memory for further fine-grained search.
   There are three chanllenges to solve when implementing SPANN:

  1. Posting length limitation: we need to bound the length od each posting list so that it can be loaded into memory in only a few disk reads, which requires us to not only partition the data into a large number of posting lists but also balance the length of posting lists.
  2. Boundary issues: the nearest neighor vectors of a query qqq may locate in boundary of multiple posting lists. Since we only search a samll number of relevant posting lists, some true neighbors of qqq that located in other posting lists will be missing[==>Commnet: multi-probe & multi-replica ]
  3. Diverse search difficulty: different queries may have different search difficulty, some queries only need to be searched in one or two posting lists while some queries require to be searched in a large number of posting lists.

Posting length limitation

  Blancing the length of lists means we need to minimize the variance of the posting length ∑i=N(∣Xi∣−∣X∣/N)2\sum_{i=}^{N}(|X_{i}| - |X|/N)^2∑i=N​(∣Xi​∣−∣X∣/N)2. To address the this problem, we can leverage the multi-constraint balanced clustering algorithm to partition the vectors evenly into multiple posting lists:

  This algorithm cannot work due to the difficulty of large N-partition balanced clustering problem and the extremly high clustering cost when ∣X∣|X|∣X∣ and NNN are very large. Therefore, we introduce a hierarchical multi-constraint balanced clustering technique, which clusters the vectors into a small number of clusters iteratively until each posting list contains limit number of vectors. What’s more, we use additional tow techniques to optimize the algorithm:

  1. Replcae the centroid with the vector that is closest to the centroid to represent each posting list through which the wasted navigating computation is transformed to the distance computation for a subset of real candidates when find the nearest posting lists for a query[===>coment: lower computation cost]
  2. In order to quickly find a smalle number of nearest posting lists for a query, we create a memory SPTAG index(space patition tree and a relative neighborhood graph) for all the vectors that represent the centroids of the posting lists.

Posting list expansion

  To deal with boundary issue, we need to increase the visibility for those vectors that are located in the boudary of the posting lists. We introduce a closure multi-cluster assignment solution for those vectors on the final clustering step–assign a vector to multile closest clusters instead of only the closest one if the distance between the vector and these clusters are nearly the same:

  However too many duplicated vectors in the close posting lists will waste the high-block disk read, for which we further optimize the closure assignment by using RNG rule to choose multiple representative clusters for the assignment of an boundary vector to reduce the similarity of two close posting lists. That is, we will skip the cluster ijijij for vector xxx if Dist(cij,x)>Dist(cij−1,cij)Dist(c_{ij},x) > Dist(c_{ij-1},c_{ij})Dist(cij​,x)>Dist(cij−1​,cij​). The insight is that two close posting lists are more likely to be both recalled by the navigating index.[===>comment: 这里的思想，和 HNSW 中启发式选择邻居的算法思想有相通之处，可以联系对比学习]

Query-aware dynamic pruning

  We introduce the query-aware dynamic pruning techinique to reduce the number of posting lists to e searched according to the distance between query and centroids instead of fixed KKK lists to be searched for all queries. A posting list to be searched only if the distance between its centroid and the query is almost the same as the distance between the query and the closest centroid, which significantly reduce the query lantency while still preserving the high recall.

Distributed Search

  Another advantage of SPANN is that the partial seach on the nearest posting lists idea can be easily extended to the distributed scenario when combined with Pyramid approach[REF-3].
  The only challenge is that there may have some hot-spot machines, for which we need to balance not only the data size but also the query access in each machine. So we partion the vectors into multiple small partitions(larger than machine number) and then use the bset-fit bin-packing algortihm to pack the small partitions into large bins(the number of bins equals to the number of machines) according to the history query access distribution.[===>comment: AI驱动、硬件加速、分布式搜索 from REF-2]
