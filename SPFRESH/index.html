SPFRESH, Incremental In-Place Update for Billion-Scale Vector Search

Overview

  As the amount of vector data grows continously, it becomes important to supprt updates to vector index. To amortize costs, most existing systems maintain a secondary index to accumulate updates(OUT-OF-PLACE-UPDATE), which are merged with the main index by globally rebuilding the entire index periodically, e.g.,Vearch[2] and Miluvs[3]. However, this approach has high fluctuations of seach latency and accuracy, not to mention that it requires substantial resources and is extremely time-consuming to rebuild.
  This paper introduces SPFRESHSPF_{RESH}SPFRESH​, which supports incremental in-place update for billion-scale vector search and implements LIRE, a Lightweight Incremental RE-balancing protocol to split overly large postings and reassign vectors across neighboring postings when necessary. In short, SPR_RESH is a disk-based and cluster-based vector index that supports lightweight incremental in-place updates without the need for global rebuild to adapt to the data distrbution shift.

Details

Cluster-based vector indices

  Cluster-based vector indices organize vector indices via clustering, where vectors in close proximity are kept in the same partition(IVF).These indices are firendly to updates because each vector insertion or deletion only involves local modifications of vector data in the corresponding partition. However, a native update on partitions may eventually lead to imbalanced clusters and consequently deteriorate search tail latency and accuracy.
  A key property of a well-partitioned vector index is the nearest partition assignment(NPA):each vector should be put into the nearest posting so that it can be represented by the posting centroid. In a cluster-based vector index, we may violate the NPA property of the index if we directly split the posting list whose length is beyond the maxinum-length-threshold.

SPANN

  LIRE is built on SPANN, a billion-scale cluster-based vector index optimized for secondary storage. It stores the vectors as a huge number of postings’ on disk, each represents a cluster of close-by vectors. Moreover, SPANN origanizes a graph-based in-memory index, SPTAG, for the centroids of all postings, to quickly identify relevant postings for a query.
  For a query, if first identifies the closet posting centroids through the in-memory index, and then loads the corresponding postings from the dis to memory for further search.

LIRE Protocol

  The core of SPFRESHSPF_{RESH}SPFRESH​ is LIRE, a Lightweight Incremental Rebalancing protocol that accumulate small vector updates of local vector partitions, and re-balances local data distribution at a low cost. LIRE is capable of maintaining index quality by fixing data distribution abnormalities locally on-the-fly and consists of the following basic operations:

  1. Insert & Delete: LIRE directly inserts a new vector to the nearest partition foloowing the original SPANN index design. It ensures the deleted vectors will not appear in the search results and will eventually be removed from the corresponding postings.
  2. Split: When a posting exceeds a length limit, LIRE evenly splits the oversized posting into two samller ones and a reassign process will be trigered for the vectors in the split postings as well as nearby postings.
  3. Merge: When a posting size is smaller than a lower threshold, LIRE identifies its nearest posting as candidates for merging. After it, a reassign process will also be trigered beacause the deletion of the old centroid might break the NPS rule after being merged with the other posting.[ we also need to prove that such split-reassignment process will always converage]
  4. Reassigning Vectors: For a merge process, only vectors from deleted posting require reassignment because the deletion of a centroid does not break NPA compliance of vectors from undeleted postings. On the other hand, a split not only deletes a centroid but also creates tow ones. Therefore a split creates more complex scenarios of potential NPA violations. So the paper drives two necessary conditions for reassignment after splitting:
      a vector vvv in the old posting with centroid AoA_{o}Ao​ is required to consider being reassigned if:

D(v,ao)≤D(v,ai),∀i∈1,2D(v,a_o) \leq D(v,a_i),\forall i \in {1,2} D(v,ao​)≤D(v,ai​),∀i∈1,2

  where D denotes the distance, AoA_{o}Ao​ represents the old centroid before spliting, and AiA_{i}Ai​ represents any of two new centroids. This condition means that if the old centroid is the closest centroid to the vector compared to new centroids then it cannot be ruled out the possibility that vvv is closer to a centroid of some nearby posting than new centroids.
  a vector vvv in the nearby posting with centroid BBB needs to consider being reassigned if:

D(v,Ai)≤D(v,Ao),∃i∈1,2D(v,A_i) \leq D(v,A_o),\exists i \in {1,2} D(v,Ai​)≤D(v,Ao​),∃i∈1,2

  This condition suggests vvv’s new beighboring centroids are getting closer than the old one. Therefore it’s necessay to check if the new and closer centroids are in fact closer than v’s existing centroid.
  According to the two conditions, a complete checking rocess would be extremely expensive. To minimize the cost, LIRE only examines neraby postings for reassignment check by selecting several AoA_{o}Ao​'s nearest postings.

SPF_RESH Design and Implementation

   SPFRESHSPF_{RESH}SPFRESH​ reuses the SPANN SPTAG index for fast posting centroid navigation as well as its searcher to serve queries and further introduces three new moldues( Upddater, Rebuilder, Block-Controller) to implete LIRE

Updater

  Updater appends a new vector at the tail of its nearest posting and maintains a new version map to keep track of vector deletion by setting a corresponding tombstone version. The map is also used yo trace the replica of each vector. By increasing the the versio number, it marks the old replicas as deleted(System keeps a global in-memory version map and store vectors along with the version number on disk. A vector is stale if the in-memory version number is greater than that on the disk). After insertions are completed inplace, the Updater check the length of the posting and then sends a split job to Local-Rebuilde if the length exceeds the split limit.

Local Rebuilder

  Local Rebuilder maintains a job queue for split(triggered by Updater), merge(triggered by Searcher when searching), and reassign(triggered by Split and Merge operation) jobs and dispatches jobs to multiple background threads.
  A Reassign job checks if vectors in the new postings and/or their neighbors need to be relocated yo re-balance the data distributions the local region based on the two necessary condition listed above. Reassigning a vector without deleting its replicas in the unexamined postings increases the replica number.[concurrent control is aslo worth learning, but not metioned here]

Block Controller

  Block Controller serves posting read, write, and append requests, as well as posting insertion and deletion operators on disk. It maintains an in-memory mapping from the posting ID to its used SSD blocks as well as the free SSD blocks pool. It offers append-only operation on postings to take advantage of the characteristics of postings and operates directly on raw SSD interfaces.[blocks management is worth learning, refer to the original paper for more details]

Crash Recovery

  SPFRESHSPF_{RESH}SPFRESH​ combines snapshot(both in-memory and on-disk data structures) and WAL for crash safety. An index snapshot is taken periodically, and update request between adjacent snapshots are collected into a WAL. The WAL will be deleted when a new snapshot is generated.
