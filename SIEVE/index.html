SIEVE, Effective Filtered Vector Search with Collection of Indexes

Overview

  Filtered vector search, is challenging as prior state-of-the-art graph-based(unfiltered) simialarity search tenchiques degenarate when hard constraints are considered: effective graph-based filtered similarity search relies on sufficient connectivity for reaching similar items within a few hops. To consider predicates, recent works propose modifying graph traversal to visit only items that satisfy predicates. However, they fail to offer the just-a-few-hops property for a wide range of predicates: they must restrict predicates significantly or lose efficiceny if only few items satisfy predicates.
  We propose an opposite approache: instead of constraining traversal, we build many indexes each serving different predicate forms. For effective construction, we devise a three-dimensional analytical model capturing relationships among index size, search time, and recall, with which we follow a workload-aware approache to pack as many useful indexes as possible into a collection. At query time, the analytical model is employed yet again to discern the one that offers the fastest search at a given recall.

Details

Preliminary

  SIEVE builds on HNSW, a graph-based vector index which combines the idea of small-world and skiip-lists to create a multi-layer graph structure for effective simiarity search over vector datasets. To serve for filtered vector search, SIEVE uses result-set-filtering in HNSW indexes as it is applicable without specialized graph construction, which may incur excessive time-to-index(TTL) and limit discussion to result-set-filtering in the following sections.(Note: SIEVE can also use ACORN’s filtering instead given minor adjutsments)

Framework Overview

  SIEVE effectively serves filtered vector queries by building and using an index collection.

SIEVE Construction

  During construction, SIEVE aims to build a colection of the most beneficial HNSW subindexes given a memory budget and traget recall on the data distribution and a historical query workload.
  Inputs. SIEVE takes as input (1) an attributed vector dataset, (2) a historical query workload–a set of query filters with probability/frequency counts, (3) a target recall, and (4) a memory budget.[ SIEVE does not restrict attribute or filter forms, only requiring filters to be evaluable on attributes]
  Cost Modeling. SIEVE models candidate indexes’ memory size and serving speed given their construction with sufficient density/M to serve queries at the target recall. It then accordingly sets up the candidates’ unit(marginal) benefits for optimizations.
  Optimization. SIEVE selects the subindexes to build under the memory budget with. greedy submodular optimization, prioritizing high-unit marginal benefit and/or high (re)use-probability subindexes in a manner akin to Materialized View Selection.
  Indexing. SIEVE builds the chosen sunindexes over the datasets. SIEVE always includes the base index over the entire vector dataset in the collection, which acts as a fallback for queries that any other subindex in the collection connot effectively handle. This design choice allows SIEVE to handle arbitrary (un-)filtered queries.

Serving Queries with SIEVE

  For serving, SIEVE aims to choose the optimal search method for fitered queries based on the subindexed in the built collection and (a potentially different from construction-time) target recall.
  Identifying the Optimal Indexed Search. The first, straightforward approach SIEVE uses for serving a query is with a built subindex. SIEVE finds the best subindex/sef combination for serving the query at target recall(preferably a small subindex in which the query is dense, using low sef)
  Choosing Search Method. SIEVE chooses between serving the query with the best-found subindex/sef combination or brute-force KNN. SIEVE estimates serving speed of both methods with its cost model, then chooses the faster one, analogous to MV-aware query rewriting.
