SQUASH, Serverless and Distributed Quantization-based Attributed Vector Similarity Search

Overview

  Recent work extends ANNS to support attribute filtering(AF), where the vector similarity is constrained with filters on additional attributes. In this paper, we introduce SQUASH, a serverless and distributed hybrid vector ANNS system designed as an elastic Function-as-a-Service(Faas) architecture, which supports advanced filtering on both continuous and catagorical attributes of any cardinality, including complex logical conditions. The following summarizes the main technical contributions of SQUASH:

  1. Design OSQ, a distributed quantization-based indexing method for both vector and attribute data. It combines multiple dimensions into shared segments for optimal compression, and enables native filtering support
  2. A low bit variant of OSQ is developed for early candidate pruning via fast bitwise comparisons, which enables high recall with low reranking requirements
  3. Design a multi-stage search pipeline including attribute filtering, partition ranking and selection, effective pruning via low-bit OSQ, a minimal set of distance calculations and an optional post-refinement step
  4. A tree-based method for synchronous Faas invocation is designed to quickly scale to thousands of concurrent instances. The multi-level approach and ID selection scheme enables the efficient coordination of parallel workers. A data retention mechanism is also intrduced to enable reuse of relevant vector indexes in runtime containers.

Optimized Scalar Quantization(OSQ)

  OSQ introduces a segment-based storage scheme which enables the theoretical compression of SQ to be realized in practice and lets us encode an appropriate number of bits for each dimension( the number of bits assigned to a dimension determines the precision of the quantized approximations). To perform the non-uniform bit allocation, bits are iterativey assigned to the dimension with the highest variance which is reduced accordingly after each assignment. Finally, the resulting bit allocations BBB and quantization cell counts C(C(C(B[j]=k \longrightarrow C[j]=2^k$) are recorded, each dimension vi,jv_{i,j}vi,j​ of vector viv_ivi​ is quantized by identifying its cell within dimension jjj.[ 不同于传统的标量压缩技巧为每个维度使用一个 unit8 来表示，这里根据数据的分布方差来分配更多的位数给该维度，进而提升精度。同时还设计了混合存储结构，避免空间浪费 ]

  In order to access individual dimensions from OSQ index, the OSQ extraction scheme utilizes column-wise lightweight left/right bit-shift operations. These operations are acclerated with vectorization, allowing a given dimension to be extracted simultaneously for all vectors. Further, in the hybrid search setting, we typically consider a reduced candidate set after filtering, that’s extraction operations are only performed on rows satisfying the predicate filter.[ 位运算获取给定维度的量化表示，过滤整体上走的 pre-filtering 策略]

Multi-Stage Search Rountine

  This section describes the multi-stage search routine in SQUASH in four funcyional areas, namely coarse partition, pre-processing, fine-grained index and post-refinement.

Coarse Partitioner, Pre-Processing

  The coarse partitioning step involves constrained clustering to extract balanced partitions for computational local balance in the resource-constrained FaaS environment. Within each partition, we design an OSQ index. We also apply an optional unitary transformation in order to decorrelate dimensional and compact energy which boosts the performance of our non-uniform bit allocation( Karhunen-Loeve Transform is used to make each partition not only more efficient and parallel, but also captures the local data distribution more accrately ). Finnaly, we perfoerm efficient one-dimensional K-means clustering to design optial scalar quantizers based on the data distribution.[ 数据分区构建分布式解决方案 ]

Filter Partition Ranking and Selection

  At query time, we fist carry out the attribute filtering workflow and gloabbly calculate the filter mask FFF, next we determine which partitions to visit to answer a given query(for all visted partitions, we consider all vectors passing the filter as candidates). Our goal is to vist the minimal set of partitions while satisfying both of the following: (1) at least kkk vectors passing the filter will be considered; (2) all partitions whose centroids are within a multiplicative distance factor TTT of the nearest(to be query) are visited.[ 先做标量前置过滤，然后再决定要访问多少个分区以及每个分区内要访问哪些向量数据 ]

Fast Pruning via Low-Bit OSQ

  Having reduce the search scope via attribute filtering and partition selection, we wish to quickly prune a significant portion of the remaining local candidates with minimal accuracy loss(partition-level) to avoid full diatance calculations. Therefore, in addition to the primary index, we also build a low-bit OSQ index(also in memory), which assigns a single bit to each dimension and can be leveraged for early candidate pruning vis fast bitwise comparisons. The low-bit OSQ scheme utilizes binary quantization, in conjunction with our segment-based storage scheme, to consolidate the 1-bit representations for S dimensions into each S-bit segment. We first standardize the data, before thresholding vectors around 0 to determine 0/1 mappings. Following this, binary values are stored into segments via OSQ.
   We compute(binary quantized) query-to-vector Hamming distances for all local candidates, and the percentage cutoff HpercH_{perc}Hperc​ can be tuned according to the approximation tolerance of the user.[ 二级索引，二进制量化，进一步实现分区排除 ]

Fine-Grained Distance Calculations

  Using the primary OSQ index, we approximate the distance from a query qqq to a given vector by identifying the distance from qqq to the nearest edge of the high-dimensional cell containing that (quantized) vector, this is a lower bound(LB) distance calculation and is calculated as the squared root of the sum, over all dimensional, of the squared distances to the nearest boundary values(ADC, 因为查询向量没有被量化). We only calculate LBs for candidates not pruned thus far, and candidates are ranked in ascending LB diatance order.
  Importantly, we note that many vectors are quantized to the same cell in a given dimension, for which we construct an in-memory ADC lookup table LLL of dimensionality (M+1,d)(M+1,d)(M+1,d) for each query qqq.[ ADC 方式计算每一个候选向量和查询向量的距离 ]

Post-Refinement and Result Aggregation

  As an optinal step, random reads to disk can be performed to fetch un-quantized vector data, and compute full-precision distance calculations. This would boost recall and improve the ordering of the final result set. When doing so, R⋅k(R>1)R·k(R\gt 1)R⋅k(R>1) records are retrived, enbaling the discovery of any true neae=rest neighbors with LB distances just outside the top-k.
  Finally, we perform an MPI-style reduce operation, where per-pertition “local results” are merged at the global level.[ 可选的精排 + 全局归并结果 ]

SQUASH Serverless Design

SQUASH Run-Time Entities

  The run-time system splits the attributed vector search task over three key components:

  • Coordinator(CO): the hub of SQUASH, it parses the user input, initiates the FaaS invocation tree, and return consolidated results
  • Query Allocatoe(QA): operates in a query parallel fashion, each QA begins by synchronously lauching a set of child QA instances. It determines its own role in the invocation tree and how many child instances it should launch based on the size and the depth of the tree. For settings involving multiple queries, it batches together the relevant queries for earch partition.
  • Query Processor(QP): one per ppartition visited, which performs the per-partition processing. Having received query metadata from its calling QA, it performs the low-bit OSQ pruning and fine-grained distance calculation steps, as well as the optional post-refinement stage. It then packages the final top-k results for its query set and restuns them to its parent QA instance.

Data Retention Exploitation

  FaaS-based solutions can benefit greatly from “warm starts” due to the reduced invocation latency it provides. We introduces Data Retention Exploitation(DRE) to enable subsequent invocations to entirely avoid external I/O assiciated with reading OSQ index files.
  DRE utilizes “singletone classes” to store key data in global areas which are retained across cloud function invocations due to container re-use. When the QA/QP handler functions are invoked, before downloading their data from object storage, they first check if the relevant data is already held in the singleton object. If so, and the dataset details match, the fect request is avoided and the global data is re-used.
   Seperately, we also developed an optinal lighweight result caching solution which saves results from earlier queries and avoids re-processing repeated requests.[ 单例模式管理数据，缓存思想 ]

Tree-based FaaS Invocation

  We present a tree-based nulti-level invocation structure, which offloads most of the invocation/response gathering workload to the QAs themselves instead of the CO. We launch multiple levels of QAs, each of which is responsible for invoking and gathing results from all QAs in the sub-tree rooted below itself.[ 分而治之，负载转移，避免单点瓶颈 ]

Task Interleaving for Multi-Query Optimization

  The QA can perform task interleaving to increase utilization while waiting for child QPs to return results, which enables an effective overlap of comnication with computation in multi-query workloads(invoke QAs/QPs via background threads). In particular, we perform attribute filtering and partition selection whilst waiting for QP responses from the previous query/batch. We then capture the per-partition results from QPs, and perform the required merge sort tp produce global results.[ 非关键步骤异步化，最大化系统吞吐量 ]
