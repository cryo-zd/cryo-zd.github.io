EnhanceGraph, A Continuously Enhanced Graph-based Index for High-dimensional ANNS

Overview

  In this paper, we address the issue that construction and search logs has not been well utilized in existing graph-based indexed for ANNS. We introduce the EnhanceGraph framework, which integrates two types of logs into a novel structure called conjugate graph. The conjugate graph is then used to improve search quality. For the search logs, the conjugate graph stores the edges from local optima to global optima to enhance routing to the nearest neighbor. For the construction logs, the conjugate graph stores the pruned edges from the proximity graph to enhance retrieving of k nearest neighbors.

Details

Framework

  We distinguish the edges of the proximity graph into two categories: routing edges and non-routing edges. Routing edges connect to the next nearest neighbor in the candidate set and are used to quickly locate the reion near the query during the search process for finding the finaly nearest neighbor. Non-routing edges do not change the next nearest neihbor in candidate set and are used to find the k-NNs. However, during the incremetal edge pruning process in the construction of proximity graph, it’s inevitable to encounter situations where these edges are not recognized.
  Utilzation of Search Log: we adopt a data-griven aproach using the search log. If we have the historical query logs, we can identify the local optimum that most queries converge to. A local optimum arises due to the absence of routing edges from the local optimum to the global optimum. If many similar queries all lead to the same local optimum, we can guarantee that all those queries can reach their global optimum by adding edges from the local optimum to the global optimum. Therefore, the search log can be utilized to continuously optimize the graph-based indexes.
  Utilization of Construction Log: to improve Recall@kRecall@kRecall@k is to increase the overlop between the graph-based indexes and the full-kNNs graph and the key issue is how to construct an approximate k-NNs graph that retains more important non-routing edges and utlizes them in the specific search phase. We can leverage the construction log to help us achieve this. During the graph construction process, each base point is repeatedly searched on the graph for its k-NNs. However, the search results are only used for pruning. Instead, we can store the pruned edges of each base point into a conjugate graph to complement the proximity graph.
  Framework: specifically, each base point is searched for its k-NNs during the constrction process. Then we insert pruned edges into the conjugate graph and insert the remaining edges into proximity graph. We use the search log from both generated and historical queries to further improve the conjugate graph. That’s, we connect the local optima with their corresponding global optima in the search log. Then, the proximity graph is used for searching a preliminary result, and the conjugate is used as an enchancement for the result.

  As shown in Algorithm 2, we utlize the conjugate graph to enhance the greedy search process. First, we search the proximity graph for preliminary nearest neighbors and the local optima(line 1-2). Next, we utilize the connections of those local-global optima pairs that are stored in the conjugate graph based on search log. It’s possible to retrieve the global optimum from the neighbors of G′[xl]G^{'}[x_{l}]G′[xl​](line 3). Once we reach the global optima of the query point, we use its neighbor G′[xg]G^{'}[x_{g}]G′[xg​] in the conjugate graph as the complement of the K-NNs(line 4-5).

Self-Generated Search Log

  In addition to utilizing historical search logs, we can also generate queries using heuristic methods. The search log from these generated queries can also continuously optimize the graph through identifying potential flaws. Thus, we first investigate which types of queries are more likely to encounter graph flaws by Voronoi Dagram.

  According to this definition, each Voronoi cell of VD(xg)V_{D}(x_{g})VD​(xg​) is a subspace in RdR^{d}Rd. Any point that lies within the subspace have xgx_{g}xg​ as its closest Voronoi site. It means that for any query within this cell, its nearest neighbor should be the point xgx_{g}xg​.
  We consider each base point xbx_{b}xb​ in the dataset DDD as a Voronoi site. In a high-dimensional space, each base point and its k-NNs are likely to be locally homogeneous.   (Heuristic) We use a heuristic to generate query points to detect flaws: points near the boundary between two adjacent Voronoi cells are more likely to encounter flaws. The k-NNs with larger rankings may fail to meet the homogeneity reqiurement.

Log-Based Search Enhancement

Optimization on the Nearest Neighbor Search

  According to Theorem IV.1 and Observation 1-3, we can infer that local optima are likely to be among the k-NNs of a base point. We can then group queries based on the local optima they converage to. Since the number of queries greatly exceeds k, many queries will converage to the same local optimum. This suggests that routing edge from the local optimum to the global optimum is missing. To address this, we can construct a conjugate graph with edges connecting local optima to their corresponding global optima. After the first-stage search concludes, the conjugate graph directs all queries that converaged to local optima towards the global optimum.
  We use two categories of search logs: the self-generated search logs and the historical search logs. Following Theorem IV.1 and observations 1-3, we generate queries for each base node to identify optential flaws. Optimization on k-NNs Search

  Some existing graph-based indexes, approximate RNG or MRNG by adopting the same edge pruning strategies to remove redundant edges. They strive to carefully select edges in the graph index to reduce the number of iterations and shrink the fan out of seach iteration. They use incremental insertion for construction, similar to the Algorithm 4. Initially, a greedy search is performed for each newly insrted point(Lines 1-2). Then, for the vistited list VVV, redundant edges are elimated based on a specific edge pruning strategy(Line 3). For example, the pruning strategies prune the longest edges of all possible triangles(RNG strategy), or ensure that the angle between any two edges is greater than 60 degrees(MRNG strategy). Finally, the remaining edges are insrted into the graph(Lines 4-7).
  During the construction process, approximate k-NNs are found for each base point(Line 2). However, this construction log is not further utilized. Only the pruned set of edges NbN_{b}Nb​ is kept for insertion into the proximity graph(Phase 1). The pruned edges may reduce the navigation efficiency, but they are valuable as non-routing edges. We store them in the conjugate graph G′G^{'}G′(Line 8). Finally, we remove redundant edges that already exist in the proximity graph to improve storage efficiency(Lines 9-10).
