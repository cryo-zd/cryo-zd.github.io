DisKANN, Fast Accurate Billion-point Nearest Neighbor Search on a Single Node

Overview

  We present a new graph-based indexing and search system called DiskANN than can index, store and search a billion point database on a single workstation with just 64GB RAM and an inexpensive SSD. In short, DiskANN is a high-quality SSD-resident index, which optimizes the algorithm to build graph index and utilizes SSD.

Details

Background

  Many applications require fast and accurate search on billions of points in Euclidean metrics. Before DiskANN, there are essentially two high-level approaches to indexing large datasets:

  1. IVF + Data Compression: Cluster the dataset into M partitions and compare the query to only the points in a few partitions closest to the query. Since the full-precision vectors cannot fit in main memory, the points are usually compresses using a quantization schema such as PQ.
  2. Sharding: divide the dataset into disjoint shards, and build an in-memory index for each shard. Queries are routed to all shards, and the results from all shards are aggregated.
       The scalability of both classes og algorithms is limited by the fact that they construct indices meant to be served from main memory. Moving these indices to disks, even SSDs, would result in a catastrophic rise of search latency and a corresponding drop in throughput. And there are two main challenges in designing a performant SSD-redsident index lie in reducing (a) the number of random SSD accesses to a few dozen, and (b) the number of round trip requests to disk under ten, preferably five.

DiskANN: Contructing SSD-Resident Indices

  1. Caching

  To futher reduce the number of disk accesses per query, we cache the data associated with a subset of vertices in DRAM, either based on a known distribution, or simply by caching all vertices that are C=3or4C=3 or 4C=3or4 hops from the starting point sss.[ 借助批量化IO读取和缓存数据的方式降低磁盘IO]

Re-Ranking

  Since PQ is a lossy compression methos, we use full-pression coordinates stored for each point next to its beighborhood on the disk. In fact, when we retieve the neighborhood of a point during search, we also retrieve the full coordinates of the point without incurring extra disk reads. Hence, as BeamSearch loads neighborhoods of the search frontier, it can also cache full-precision coordinates of all the nodes vistied during the search process, using no extra reads to the SSD, which allows us to return the top kkk candidates based on the full precision vectors.
